Runbook 07 — Custom fixtures (UI + API) + composition rules

Objective

Establish a single, documented fixture contract so tests read like business workflows. Sprint 7 standardizes: (1) UI auth via storageState, (2) API auth via an API client fixture, (3) Page Objects as domain primitives, and (4) composition rules (what is worker vs test scope).

Verified auth contract One fixture entrypoint

Contract: UI auth is created once in globalSetup and stored in .auth/storageState.json. Logged-out state is always available via .auth/emptyState.json. Tests select auth state by choosing the Playwright project: auth-chromium or unauth-chromium.

What “done” means

  • ✔ Tests import from a single entrypoint: const { test, expect } = require('../fixtures/rw').
  • ✔ UI auth is deterministic via globalSetup + storageState (no token injection “side path”).
  • ✔ API auth is deterministic via api fixture that logs in and stores JWT in the client.
  • ✔ Page Objects are provided by fixtures (home, editor, profile, etc.).
  • ✔ Scope rules are documented (what runs once vs per-test, what is safe to share).

Canonical auth strategy (choose one)

Selected for this lab StorageState + globalSetup (matches your playwright.config.js)
Why It is CI-friendly, fast (login once), and keeps tests clean (no per-test UI login).
What to avoid Competing auth strategies (e.g., localStorage token injection) living alongside storageState.

Rule: one blessed UI auth path. If another auth approach exists in the repo, it must be clearly marked legacy or removed, so new hires don’t “pick a random way.”

Fixture contract (what tests are allowed to assume)

From fixtures/rw.js (canonical):

  • api — authenticated API client with JWT set (via /users/login).
  • home, auth, editor, article, profile, settings — Page Objects bound to the current page.

Composition rules

  • Tests do not fetch env vars. Fixtures own env parsing (getEnv()).
  • Tests do not login. Logged-in UI is a project selection; API auth is the api fixture.
  • Tests do not “newContext” ad-hoc. If a test needs a different state, create a new project or a new fixture.
  • Pages do mechanics; tests describe intent. (Page Objects are the API.)

Scope guidance (worker vs test)

Worker-scoped (runs once per worker) globalSetup (UI login + write storageState), anything expensive & shareable, environment contract verification.
Test-scoped (runs per test) Page Object instances bound to page, API clients bound to request, test-specific data creation, assertions, and evidence capture.
Anti-pattern Hidden “once per suite” state inside tests (order dependency).

Interpretation for your lab: globalSetup proves auth once. Each test should remain independently runnable, with state seeded via API when needed (Sprint 5), and UI workflows tested via Page Objects (Sprint 8).

Commands

Required env vars (your suite enforces these):

export WEB_BASE_URL="http://sut.testlab:3000"
export API_BASE_URL="http://sut.testlab:3001/api"
export RW_USER_EMAIL="rwuser_pw@yourdomain.test"
export RW_USER_PASSWORD="yourpassword"

Run a logged-in test (uses storageState):

npx playwright test specs/sprint7 --project=auth-chromium --workers=1 --reporter=line

Run a logged-out test (forces unauth state):

npx playwright test specs/sprint7 --project=unauth-chromium --workers=1 --reporter=line

Skip globalSetup (useful for debugging auth failures fast):

SKIP_GLOBAL_SETUP=1 npx playwright test specs/sprint7 --project=unauth-chromium --workers=1 --reporter=line

Expected outputs

  • ✔ .auth/emptyState.json exists before tests start (no races).
  • ✔ In auth project, navbar shows authenticated items without UI login steps in the test.
  • ✔ API fixture can seed state (create article, fetch article) without UI.
  • ✔ Failures have trace/video/screenshot in test-results/ and HTML report in playwright-report/.

Failure modes and diagnosis

  • Auth state missing: .auth/storageState.json not present → run a single test with globalSetup enabled; check [globalSetup] logs and the saved screenshot (global-setup-failure.png).
  • Wrong project selected: tests behave logged-out in an auth flow → ensure you ran --project=auth-chromium.
  • Conflicting auth strategy: token injection code changes behavior → remove/rename legacy fixtures so only storageState is “the way.”
  • Env contract broken: hard fail at startup → confirm WEB_BASE_URL, API_BASE_URL, RW_USER_EMAIL, RW_USER_PASSWORD.
  • API auth fails: /users/login non-200 → capture status/body in the API client error; confirm API base URL includes /api.

Rule: do not “fix” these by waiting longer. Fix the contract (env/auth/project selection) so the suite stays deterministic.

Why this matters (production relevance)

Without a single fixture contract, teams drift: multiple login strategies, inconsistent environment reads, and tests that are impossible to reason about. Sprint 7 makes the suite scalable by ensuring every test begins from a known state, uses shared domain primitives, and produces reliable evidence.

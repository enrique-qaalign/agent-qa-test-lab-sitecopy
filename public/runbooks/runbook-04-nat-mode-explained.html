Sprint 4 Addendum — Privacy-Safe CI Mode

Documents the intentional, temporary operating mode used during Sprint 4 recordings: keep the SUT isolated behind VirtualBox NAT + port forwarding, and reach it from a separate CI runner via an explicit SSH tunnel. This protects private network details while preserving real network boundaries.

NAT isolation loopback-only ports SSH tunnel bridge no LAN identifiers on video diagnosable failures

Objective + success criteria

Objective: enable Sprint 4 video recordings and public artifacts without disclosing LAN IP ranges, hostnames, or internal topology, while maintaining a real runner → SUT network boundary.

Success criteria:

  • All logs and recordings reference 127.0.0.1 only (no LAN IPs or private hostnames)
  • CI runner and SUT remain on separate machines (no localhost illusions)
  • Connectivity is explicit (tunnel is required; failures are fast and diagnosable)
  • SUT remains isolated behind NAT until Sprint 4 recording is complete

Topology (Sprint 4)

Old Mac mini Hosts the SUT: VirtualBox VM + Docker Compose (Conduit). Networking is NAT. Access is via port forwarding bound to loopback.
New Mac mini / MacBook Pro Hosts the CI runner only. No VirtualBox. Must reach the SUT via an SSH tunnel (runner → old-mac → VM).
Key principle CI does not “guess” networks. CI uses a declared contract and fails fast when the contract is broken.

This mode is temporary: after Sprint 4 recording, the VM will switch to bridged networking.

Network contract

The recording-safe contract is loopback-only. Port forwarding is configured on the SUT host (old Mac mini), and the runner consumes the same endpoints through the tunnel.

API http://127.0.0.1:3001/api
Web http://127.0.0.1:3000
SSH (VM access) ssh -p 2222 sut@127.0.0.1 (on the SUT host)

Note: 127.0.0.1 here is “local” to whichever machine is consuming the contract. The tunnel makes the contract true on the runner without exposing LAN details.

Exact commands

Start SUT services (inside the VM)

cd ~/conduit
```

docker compose up -d
docker compose ps
```

Open the tunnel (run on the CI runner)

The runner must bridge to the old Mac mini (SUT host). Keep this tunnel open for the duration of the run. 

ssh -N \
```

-L 3001:127.0.0.1:3001 
-L 3000:127.0.0.1:3000 
-p 2222 sut@
```

Runner preflight (fast fail)

nc -z 127.0.0.1 3001 || (echo "API tunnel down" && exit 2)
```

nc -z 127.0.0.1 3000 || (echo "WEB tunnel down" && exit 2)

curl -fsS --max-time 3 [http://127.0.0.1:3001/api/tags](http://127.0.0.1:3001/api/tags) >/dev/null
curl -fsS --max-time 3 [http://127.0.0.1:3000/](http://127.0.0.1:3000/) >/dev/null
```

CI environment variables

API_BASE_URL=http://127.0.0.1:3001/api
```

WEB_BASE_URL=[http://127.0.0.1:3000](http://127.0.0.1:3000)
```

During Sprint 4 recording, prefer 127.0.0.1 endpoints to keep artifacts and logs sanitized.

Failure modes + how to diagnose

Tunnel not established (most common)

Symptom: runner shows connection refused to 127.0.0.1:3000/3001.

Diagnosis: verify tunnel process is running and forwarding the correct ports.

ps aux | grep "ssh -N" | grep -v grep || true
```

nc -vz 127.0.0.1 3001
nc -vz 127.0.0.1 3000

Action: re-run the tunnel command and keep it open for the duration of the job.

```

SUT not running in the VM

Symptom: tunnel is up but HTTP checks fail (5xx/connection errors).

Diagnosis: SSH to VM and confirm containers are healthy.

ssh -p 2222 sut@127.0.0.1
```

cd ~/conduit
docker compose ps
curl -fsS [http://127.0.0.1:3001/api/tags](http://127.0.0.1:3001/api/tags) | head
curl -fsS [http://127.0.0.1:3000/](http://127.0.0.1:3000/) | head

Action: start services with docker compose up -d and re-check.

```

Name resolves to LAN IP (leaks topology + breaks NAT model)

Symptom: CI tries sut.testlab and times out to a LAN address.

Diagnosis: inspect name resolution on the runner.

dscacheutil -q host -a name sut.testlab || true
```

# /etc/hosts entry may point to a private LAN IP — avoid during Sprint 4
```

Action: use loopback contract (127.0.0.1) for Sprint 4 recordings.

Why it matters (interview framing)

This mode demonstrates mature quality engineering under real constraints:

  • Security posture: prevents accidental disclosure of internal network details during demos.
  • Explicit contracts: CI uses declared endpoints and fails fast when connectivity is broken.
  • Real boundary: runner and SUT remain separate systems; the tunnel is the controlled bridge.
  • Diagnosability: timeouts vs refused connections map to specific layers and corrective actions.

This mirrors production access patterns (bastions, VPNs, SSH tunnels) where systems are reachable only through controlled entry points.

Exit criteria (post Sprint 4)

After Sprint 4 recording is complete, switch the VM from NAT to bridged networking to enable LAN-routable CI without tunnels.

  • Change VM network adapter: NAT → Bridged
  • Expose SUT endpoints on the LAN (no loopback-only constraint)
  • Update name resolution (e.g., sut.testlab) to the SUT host LAN IP
  • Remove the SSH tunnel requirement from CI

This transition is deliberate and documented to keep recordings sanitized and the lab realistic when privacy constraints are lifted.

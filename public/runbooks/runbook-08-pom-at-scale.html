Objective

Establish a scalable Page Object Model (POM) architecture so UI automation stays maintainable as the test suite grows. This runbook defines folder structure, locator standards, and a “new page recipe” that a new hire can follow without tribal knowledge.

Maintainability Locator policy Flows

Rule: tests should describe workflows; pages/components/flows own mechanics.

What “done” means

  • ✔ Standard structure exists: pages/, components/, flows/, assertions/.
  • ✔ Locator policy is documented and used (role/label/testid first).
  • ✔ A “new page recipe” spec proves how to add a page cleanly.
  • ✔ Repeated sequences moved into flows; repeated expectations moved into assertion helpers.

Recommended structure

pages/
  HomePage.*
  LoginPage.*
  SettingsPage.*
  EditorPage.*
  ArticlePage.*
  ProfilePage.*

components/
  Navbar.*
  ArticlePreview.*
  TagList.*

flows/
  AuthFlow.*
  PublishArticleFlow.*
  FollowUserFlow.*

assertions/
  expectLoggedIn.*
  expectArticleVisible.*
  expectToast.*

helpers/
  env.*
  urls.*
  dataFactories.*

If your framework already has most of this, Sprint 8 is about standardizing the “shape” and documenting how to extend it without drift.

Locator standards (guardrails)

Prefer getByRole, getByLabel, getByTestId
Avoid brittle CSS chains, deep DOM traversal, index selectors, text-only selectors that drift
Policy if you must use CSS/XPath, document why and capture a stable “anchor” element

Anti-pattern: fixing flake by “waiting longer.” Sprint 8 is about stable access and stable workflows.

Page Object contract

  • Each page has open() and assertAt() (or isAt() + explicit expect).
  • Locators are grouped and named (no “mystery locators” inline in tests).
  • Actions do actions; assertion helpers do assertions (avoid mixing unless intentional).
  • Pages should not “know” test data strategy; tests/fixtures do.

Flows (multi-page workflows)

Flows encapsulate multi-step business tasks (publish article, follow user, update settings). This keeps tests short and makes changes localized when UI steps shift.

New page recipe (the “30-minute new hire path”)

  1. Create pages/NewFeaturePage.* with open() and assertAt()
  2. If UI repeats elsewhere, factor a reusable components/ object
  3. If workflow crosses pages, create/extend a flows/ helper
  4. Add assertions to assertions/ rather than repeating inline expects
  5. Add a single smoke spec proving the recipe works end-to-end

Definition of maintainable: a UI change updates 1–2 page/component/flow files, not 30 tests.

Commands

Example recipe spec run:

export BASE_URL="http://sut.testlab:3000"
export API_BASE_URL="http://sut.testlab:3001/api"

npx playwright test specs/sprint8/Sprint8.pom.scale.spec.* --project=chromium --workers=1 --reporter=line

Expected outputs

  • ✔ Recipe spec passes and is readable (workflow-level, minimal glue)
  • ✔ Minimal repeated locator usage in tests (locators live in pages/components)
  • ✔ Any failure produces trace/video/screenshot sufficient to debug

Failure modes and diagnosis

  • “God page” objects: one class grows endlessly → split into components + flows.
  • Selector churn: locators depend on DOM structure → shift to role/label/testid.
  • Assertions scattered: inconsistent semantics → centralize in assertions/.
  • Flake reintroduced via waits: replace with stable state checks and trace-first debugging.

Why this matters (production relevance)

At scale, UI tests fail not because Playwright is weak, but because structure collapses: duplicated locators, inconsistent flows, and “smart waits” that hide real bugs. Sprint 8 proves you can keep a UI suite coherent as it grows.

Runbook 06 — Release gates (time-to-signal) + decision rubric

Objective

Sprint 6 proves a decision-grade gating model: tests map to risk, not coverage vanity. The suite is split into SMOKE (fast time-to-signal) and REGRESSION (deeper confidence) with explicit go/no-go criteria.

Time-to-signal Release gate contract

Contract: Every gate declares capability, persona, failureMeans, releaseDecision (BLOCK/WARN), mitigation, and owner. If a gate fails, CI output must be sufficient to route ownership without reruns.

Acceptance criteria (Sprint 6)

  • ✔ SMOKE provides fast signal (auth + core flows) and is runnable alone.
  • ✔ REGRESSION covers deeper workflows and cross-channel consistency (API ⇄ UI).
  • ✔ Each test is a release gate with an explicit decision: BLOCK or WARN.
  • ✔ Gate failures print a human-readable contract in CI logs (no archaeology).
  • ✔ Evidence exists: traces/video/screenshot/HTML report are retained for failures.

Decision rubric (go / no-go)

Decision Meaning Release action
BLOCK Core capability is broken or risk is unacceptable (auth, publish, read, security boundary). Do not release. Fix or rollback. Re-run gate once after change.
WARN Non-critical capability is degraded (engagement/social/optional UX) with mitigations available. Release allowed with explicit note + follow-up ticket (and/or feature flag mitigation).

Rule: BLOCK and WARN are not “severity vibes.” They are a release policy. If the mitigation is “we can’t mitigate,” the decision is probably BLOCK.

Gate anatomy (what every gate must include)

Each gate uses the wrapper in helpers/releaseGate.js to enforce the contract and annotate results.

test(
  "RW.SMOKE.001 — Login establishes authenticated session",
  releaseGate(
    {
      id: "RW.SMOKE.001",
      capability: "Authentication: establish session",
      persona: "Registered user",
      failureMeans: "Users cannot authenticate; authenticated journeys are blocked",
      releaseDecision: "BLOCK",
      mitigation: "Rollback auth change; validate /api/users/login + storageState generation",
      owner: "Web + API",
    },
    ["home"],
    async ({ home }) => {
      await home.open();
      await home.expectLoggedInNav();
    }
  )
);

Verified release gate contract: if any required field is missing, the wrapper hard-fails early so no “half-documented” gates make it into the suite.

Suite structure: SMOKE vs REGRESSION

SMOKE Fast, business-critical workflows. Goal is time-to-signal. Examples: auth session, feed renders, publish, read by slug, comment, logout.
REGRESSION Deeper workflows and consistency checks. Examples: edit persists after refresh, settings persist, API→UI visibility, UI→API visibility, unauth write blocked.
Policy SMOKE failures typically BLOCK. REGRESSION can include WARN gates where mitigation exists.

How to run (local and CI)

Env contract (preferred canonical variables):

export WEB_BASE_URL="http://sut.testlab:3000"
export API_BASE_URL="http://sut.testlab:3001/api"
export RW_USER_EMAIL="rwuser_pw@yourdomain.test"
export RW_USER_PASSWORD="yourpassword"

Run SMOKE gate suite:

npx playwright test specs/sprint6/Sprint6.release-gates.smoke.spec.js \
  --project=auth-chromium --workers=1 --reporter=line

Run REGRESSION gate suite:

npx playwright test specs/sprint6/Sprint6.release-gates.regression.spec.js \
  --project=auth-chromium --workers=1 --reporter=line

Run everything in Sprint 6 folder:

npx playwright test specs/sprint6 \
  --project=auth-chromium --workers=1 --reporter=line

Note: Your Playwright projects are auth-chromium and unauth-chromium. Do not run --project=chromium (it does not exist in your config).

Expected outputs

  • ✔ Each failing gate prints a “RELEASE GATE FAILURE CONTRACT” block with fields populated.
  • ✔ Test annotations contain capability/persona/decision/owner for reporting.
  • ✔ Artifacts exist for failures (trace.zip, screenshot, optional video).
  • ✔ You can decide go/no-go from logs + artifacts without rerun.

Failure modes and diagnosis

  • Gate fails but contract missing: wrapper not used or required fields absent → fix the test to go through releaseGate(...).
  • Auth-related gate fails: storageState not generated or credentials invalid → validate fixtures/global-setup.js and RW_USER_*.
  • API seeded gate fails: /users/login or API base URL wrong → confirm API_BASE_URL ends with /api.
  • Unauth gate flaky: test accidentally runs under auth state → ensure it uses unauth-chromium project or overrides storageState explicitly.

Rule: Don’t “fix” gate failures by adding waits. Fix the contract (auth, environment, data setup, or routing) so time-to-signal stays fast.

Why it matters (production relevance)

Release gates are only valuable if they are fast, trusted, and decision-grade. Sprint 6 makes CI a release signal by encoding business risk into each test’s contract, routing ownership cleanly, and producing evidence without rerun dependency.

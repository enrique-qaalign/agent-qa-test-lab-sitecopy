Objective

Prove that test runs are repeatable, idempotent, and parallel-friendly by controlling data and state explicitly. Sprint 5 standardizes a pattern: seed state via API, then validate UI reads without hidden dependencies on earlier UI steps.

Verified Auth Contract

All Sprint 5 tests rely on an explicit, verified authentication contract.

  • API state seeding uses a dedicated api fixture that authenticates via /users/login and injects a JWT into every request.
  • UI navigation uses Playwright storage state generated during global-setup, not ad-hoc login steps inside tests.
  • No test depends on another test’s execution, UI flow, or runtime state.

Invariant: If an API call returns 401 or 403, the failure indicates an environment or credential issue — not a flaky test.

Idempotency Parallel safety API seeding

Rule: every test must be safe to run alone, twice, or in parallel.

What “done” means

  • ✔ Tests create their own data (no reliance on “what’s already there”).
  • ✔ Re-running Sprint 5 produces new independent records (no collisions).
  • ✔ UI tests can validate API-seeded state (no UI setup flows required).
  • ✔ Suite remains stable under CI constraints (workers=1 today; design supports >1 later).

Invariants (non-negotiable)

Unique test data Use unique names/IDs per run to prevent collisions (e.g., timestamp + random suffix).
No ordering dependency Any test can run first; no “test A must run before test B”.
State created via API Prefer API seeding for speed and determinism; UI validates behavior, not setup mechanics.
Auth is standardized Auth state is established once via globalSetup and reused via storageState.

In this framework, uniqueness is provided by helpers/testData.js via Date.now() + a random suffix. That is intentional: collisions are the enemy of repeatability.

How this framework achieves repeatability

1) Environment is mandatory (no hidden defaults)

playwright.config.js hard-fails if API_BASE_URL and WEB_BASE_URL/BASE_URL are missing. This prevents “works on my machine” drift.

2) Auth is created once (globalSetup) and reused

fixtures/global-setup.js verifies credentials via API, then performs UI login and writes .auth/storageState.json. Unauthenticated flows use .auth/emptyState.json.

3) Test data is unique by default

helpers/testData.js generates unique article data on demand. This prevents collisions when tests re-run or multiple tests create articles in the same suite run.

4) API seeds state; UI proves it renders correctly

Sprint 5 uses the API client (helpers/apiClient.js) to create an article, then navigates via UI to confirm the article appears and can be opened.

Key idea

UI tests should validate the product, not waste time performing setup steps that the API can do deterministically.

Commands

Set the canonical environment variables:

export WEB_BASE_URL="http://sut.testlab:3000"
export API_BASE_URL="http://sut.testlab:3001/api"
export RW_USER_EMAIL="YOUR_USER_EMAIL"
export RW_USER_PASSWORD="YOUR_USER_PASSWORD"

Run Sprint 5 only:

npx playwright test specs/sprint5 --project=auth-chromium --workers=1 --reporter=line

Run a single Sprint 5 test by grep:

npx playwright test specs/sprint5 --project=auth-chromium -g "S5.003" --workers=1 --reporter=line

Tip: Sprint 5 should be safe to run multiple times back-to-back. That is the point.

Expected outputs

  • ✔ S5.001 creates an article via API and navigates to it successfully.
  • ✔ S5.002 creates two independent articles (distinct slugs).
  • ✔ S5.003 shows the UI can render an API-created article in Global Feed and open it.
  • ✔ Failures generate trace/video/screenshot because use.trace/video/screenshot are configured retain-on-failure.

Evidence lives in test-results/ and the HTML report in playwright-report/. The suite is designed so failures are diagnosable without rerunning.

Failure modes and diagnosis

  • API_BASE_URL wrong / missing: suite fails fast in config → confirm API_BASE_URL points to .../api and is reachable.
  • Auth fails: globalSetup fails early → check RW_USER_EMAIL/RW_USER_PASSWORD and API login endpoint /users/login.
  • Global Feed doesn’t show seeded article: UI test fails at link visibility → verify you are on “Global Feed”, and confirm the backend indexes/publishes immediately. (If the app is eventually consistent, add a principled wait via a UI state change or API confirmation—not timeouts.)
  • Hidden dependency detected: a UI test needs prior UI steps → move setup into API seeding or fixtures; keep tests workflow-focused.
  • Data collisions: duplicates overwrite/confuse expectations → ensure all created data uses unique suffixes (Sprint 5 requires it).

Anti-pattern: “fixing” state issues by adding sleeps. Sprint 5 is about determinism: seed with API, assert with UI, debug with artifacts.

Why this matters (production relevance)

In real CI/CD, the biggest cause of flaky or untrusted automation is uncontrolled state: tests that assume previous tests ran, environments that retain data, or suites that can’t be re-run reliably. Sprint 5 proves the suite behaves like production-grade automation: independent tests, deterministic setup, and repeatable results.

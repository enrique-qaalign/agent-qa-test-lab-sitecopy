Runbook 01 — Runner → SUT connectivity and naming

Validation of real runner-to-SUT network boundary and stable naming. Localhost access must not be possible.

localhost negative proof remote HTTP positive proof TCP transport proof routing proof

Objective + success criteria

Objective: Validate that automated tests execute from an isolated runner and reach the System Under Test (SUT) exclusively over a real network boundary using stable naming. “Localhost” must not be an acceptable target.

Success criteria:

  • Localhost negative proof: Requests to http://localhost:${SUT_PORT} on the runner do not return the SUT application (connection refused/timeout is acceptable; 200 OK is a failure signal).
  • Remote positive proof: Requests to ${BASE_URL} return an HTTP response from the SUT (example: HTTP/1.1 200 OK).
  • DNS proof: sut.testlab resolves to the expected LAN IP.
  • TCP transport proof: TCP handshake to ${SUT_IP}:${SUT_PORT} succeeds.
  • Automation proof: A Playwright test passes when targeting ${BASE_URL}.

Exact commands

Environment variables (runner)

export SUT_IP="192.168.0.31"
      export SUT_PORT="3001"
      export SUT_HOST="sut.testlab"
      export BASE_URL="http://${SUT_HOST}:${SUT_PORT}"

Confirm values

echo "SUT_IP=$SUT_IP"
      echo "SUT_PORT=$SUT_PORT"
      echo "SUT_HOST=$SUT_HOST"
      echo "BASE_URL=$BASE_URL"

DNS resolution proof (runner)

getent hosts "${SUT_HOST}" || nslookup "${SUT_HOST}" || dig +short "${SUT_HOST}"

Localhost negative proof (runner)

# Success = connection refused / timeout.
      # Failure = HTTP response (means something is listening locally on the runner).
      curl -sv --max-time 2 "http://localhost:${SUT_PORT}/" | head -n 12

Remote HTTP positive proof (runner)

# Your real observed proof:
      curl -sS -I "${BASE_URL}/" | head -n 1
      
      # Optional: show headers for visibility
      curl -sS -o /dev/null -D - --max-time 3 "${BASE_URL}/" | head -n 12

TCP transport validation (runner)

# macOS-friendly flags:
      nc -vz -w 2 "${SUT_IP}" "${SUT_PORT}"

Network routing validation (runner, macOS)

route -n get "${SUT_IP}" | egrep 'destination|gateway|interface'

Sprint 1 demo proof (runner)

# 1) Boundary proof
      export BASE_URL="http://sut.testlab:3001"
      curl -sS -I "$BASE_URL/" | head -n 1
      
      # 2) Automation proof

Failure modes + how to diagnose

Localhost succeeds unexpectedly

Symptom: curl http://localhost:${SUT_PORT} returns an HTTP response.

Diagnosis: A service is running locally on the runner, or forwarding/proxying is masking the boundary.

Remote hostname does not resolve

Symptom: hostname resolution fails or connects to an unintended host.

Diagnosis:

nslookup sut.testlab

Action: Correct hostname mapping (e.g., /etc/hosts) to the intended SUT IP.

TCP connection fails

Symptom: nc reports timeout or connection refused.

Diagnosis: service is down, port is not exposed, or traffic is blocked.

Action: Verify service state on the SUT, confirm port exposure, and review firewall controls.

Service bound to loopback on SUT

Symptom: service responds locally on the SUT but is unreachable from the runner.

Diagnosis: service is listening on 127.0.0.1:${SUT_PORT} only.

Action: bind to 0.0.0.0 or the LAN interface and restart the service.

Why it matters (production relevance)

This validation eliminates false confidence caused by localhost coupling and shared execution environments. By enforcing a real network boundary early, subsequent automation results reflect deployment conditions: name resolution, routing, port exposure, and service availability.

Failures observed after this point represent genuine infrastructure or application issues rather than test-environment artifacts, improving diagnosability, change safety, and release confidence.

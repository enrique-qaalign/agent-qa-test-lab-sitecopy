Runbook 01.1 — CI Runner → SUT connectivity and naming (CI Variant)

Prove CI-executed automation reaches the SUT across a real network boundary using stable naming. Localhost HTTP must not be an acceptable target. This runbook is CI-safe, fail-fast, and evidence-friendly.

name resolution proof localhost HTTP negative proof TCP transport proof remote identity proof automation proof sanitized CI evidence

Objective + success criteria

Objective: Validate that tests executed by a CI runner reach the System Under Test (SUT) over a real network boundary using stable naming (e.g., sut.testlab). Localhost HTTP must not be accepted because it can mask routing, name resolution, firewall, and port exposure failures.

Rule: HTTP checks and Playwright UI navigation must target ${WEB_BASE_URL} resolved by name (not localhost).

Compatibility note: Runbook 00 uses localhost port-forwarding for SSH baseline access (127.0.0.1:2222 → VM:22). That is acceptable for bootstrap access. This runbook prohibits localhost for SUT HTTP (web/API) proof.

Success criteria:

  • Name resolution proof: ${SUT_HOST} resolves inside the CI job to the intended SUT address (DNS or injected hosts mapping).
  • Localhost HTTP negative proof: requests to http://localhost:${WEB_PORT} do not return SUT HTTP (curl failure is acceptable; 200 OK is a failure).
  • TCP transport proof: TCP handshake to ${SUT_HOST}:${WEB_PORT} and ${SUT_HOST}:${API_PORT} succeeds.
  • Remote identity proof: web and API responses match Conduit (HTML title for web; JSON shape for API).
  • Automation proof: at least one Playwright UI test passes while targeting ${WEB_BASE_URL}.

New hire guide — how environment selection works

If you are new to the project: most “mystery failures” in automation are not code failures—they are environment selection failures. This framework supports multiple environments via environment files. Your job is to pick the right environment file for the run you are executing.

What you are selecting

TEST_ENVHuman-readable environment label (dev, qa, prod).
WEB_BASE_URLWeb UI entrypoint used by Playwright navigation. Example: http://sut.testlab:3000.
API_BASE_URLAPI entrypoint used by API setup/login. Example: http://sut.testlab:3001/api.
RW_USER_EMAIL, RW_USER_PASSWORDOptional credentials used by global setup (if enabled).
Legacy compatibilityBASE_URL may exist for older scripts. Prefer WEB_BASE_URL.

How to choose an environment (recommended patterns)

Recommended: use dotenvx to load an explicit env file per run. This mirrors CI behavior and avoids “my terminal had old variables.” 

# Dev
dotenvx run -f .env.dev -- npx playwright test

# QA
dotenvx run -f .env.qa -- npx playwright test

# Prod (example)
dotenvx run -f .env.prod -- npx playwright test

How to confirm you loaded the right environment

dotenvx run -f .env.qa -- bash -lc 'echo "TEST_ENV=$TEST_ENV"; echo "WEB_BASE_URL=$WEB_BASE_URL"; echo "API_BASE_URL=$API_BASE_URL"'

Optional: load an environment into your current shell (use carefully)

This approach is convenient, but it can cause confusion if you forget what is currently loaded. Prefer dotenvx run -f ... for repeatability. 

# zsh / bash
set -a
source .env.qa
set +a

echo "TEST_ENV=$TEST_ENV"
echo "WEB_BASE_URL=$WEB_BASE_URL"
echo "API_BASE_URL=$API_BASE_URL"

Dependencies

Required prerequisites

  • SUT is running and services are bound to a reachable interface (not loopback-only).
  • CI runner network can reach the SUT (routed LAN/VPN, correct runner tags, firewall rules in place).
  • Stable naming is configured for the SUT hostname (DNS or hosts injection).
  • No port-forwards that make SUT web/api reachable via localhost in CI.

Proof blocks (CI, fail-fast)

These blocks are written to match the actual environment contract (WEB_BASE_URL, API_BASE_URL). Where we need SUT_HOST or ports, we derive them deterministically.

0) Derive SUT_HOST + ports (CI)

# Derive host from WEB_BASE_URL
SUT_HOST="$(echo "$WEB_BASE_URL" | sed -E 's#^https?://([^:/]+).*#\1#')"
WEB_PORT="3000"
API_PORT="3001"

echo "SUT_HOST=${SUT_HOST}"
echo "WEB_PORT=${WEB_PORT}"
echo "API_PORT=${API_PORT}"

1) Name resolution proof (CI)

Proves the CI runner can resolve the SUT hostname.

getent hosts "${SUT_HOST}" \
  || nslookup "${SUT_HOST}" \
  || dig +short "${SUT_HOST}"

2) Localhost HTTP negative proof (CI)

Deterministic: FAIL if localhost returns HTTP for the web port. PASS if connection refused/timeout. 

curl -fsS --max-time 2 "http://localhost:${WEB_PORT}/" >/dev/null \
  && { echo "FAIL: localhost returned HTTP (boundary bypass)"; exit 1; } \
  || echo "PASS: localhost did not respond (boundary enforced)"
Private (operator): local listener diagnostics (if localhost succeeds) 
# If localhost unexpectedly succeeds, identify the listener.
# Linux:
ss -lntp | egrep ":(3000|3001)\b" || true

# macOS runner host:
lsof -nP -iTCP:3000 -sTCP:LISTEN || true
lsof -nP -iTCP:3001 -sTCP:LISTEN || true

3) TCP transport proof (CI)

Success condition: exit code 0 for both handshakes.

nc -vz -w 2 "${SUT_HOST}" "${WEB_PORT}"; echo "exit_code=$?"
nc -vz -w 2 "${SUT_HOST}" "${API_PORT}"; echo "exit_code=$?"

4) Remote identity proof (CI)

Web UI identity (Conduit)

curl -sS -I "${WEB_BASE_URL}/" | head -n 1
curl -sS "${WEB_BASE_URL}/" | grep -i "<title" | head -n 1

API identity (Conduit)

curl -sS -I "${API_BASE_URL}/tags" | head -n 1
curl -sS "${API_BASE_URL}/tags" | head -c 200

5) Automation proof (CI)

Replace the spec path/tag with your actual Sprint 1 connectivity spec. Contract: load ${WEB_BASE_URL} and assert a Conduit UI signature (e.g., navbar contains “conduit”).

npx playwright test specs/sprint1/Sprint1.runner-to-sut.connectivity.spec.js \
  -g "RW.NET.001" \

Failure modes + diagnosis

Environment not loaded (variables are blank)

Symptom: WEB_BASE_URL or API_BASE_URL is empty in logs.

Likely cause: you ran without dotenvx run -f .env.<env> or without sourcing an env file.

Action: re-run using the explicit env file load pattern.

Name resolution fails in CI

Symptom: getent/nslookup/dig returns nothing or errors.

Likely cause: CI runner not using the same DNS/hosts mapping as local runner.

Action: inject hosts mapping at runner level or move job to a runner with correct DNS/VPN.

Localhost HTTP negative proof fails (localhost returns HTTP)

Symptom: localhost responds with 200/30x.

Likely cause: port-forward/proxy/local listener is bypassing the intended boundary.

Action: identify the listener, disable port-forwards, and re-run.

TCP handshake fails

Symptom: nc times out or refuses.

Likely cause: SUT service down, port not exposed, or firewall segmentation.

Action: confirm SUT services are healthy and bound to a reachable interface; verify runner routing/firewall.

HTTP succeeds but identity is wrong

Symptom: status line is OK, but title/JSON does not match Conduit.

Likely cause: wrong host mapping, proxy/captive portal, or wrong port routing.

Action: re-check name resolution source and validate SUT port mappings.

Why it matters (production relevance)

CI is the enforcement point for release confidence. If CI can only pass by targeting localhost, the pipeline is validating a coupled execution environment rather than deployment reality. This runbook forces proof of name resolution, routing, and port exposure—exactly the surfaces that fail in production.

When this runbook passes, subsequent failures are far more likely to represent genuine application or infrastructure issues rather than “works on my machine” artifacts. That improves diagnosability, change safety, and gate credibility.

Public vs operator: Hiding operator blocks in HTML is not secrecy. If an artifact contains LAN IPs, key fingerprints, internal hostnames, or tokens, store it in a private repo and/or private CI artifacts.

Runbook 00

Golden VM baseline — Ubuntu Server + SSH keys + snapshot

This runbook establishes a repeatable baseline VM for the lab: a clean Ubuntu Server install, patched to current updates, reachable via SSH using keys, with a snapshot used as the permanent rollback point.

Deterministic access Known-good rollback Repeatable rebuild Lab-safe defaults

Objective

  • Create a clean Ubuntu Server VM suitable for lab/SUT use.
  • Enable SSH and transition to key-based authentication.
  • Patch the OS to current updates.
  • Snapshot the baseline before installing any “real stack” services.

Success criteria

  • Passwordless SSH: ssh -p 2222 sut@127.0.0.1
  • System patched: sudo apt update && sudo apt upgrade -y
  • Snapshot exists: baseline-ubuntu-sshkeys

Procedure

1) VirtualBox VM creation

  • Select the Ubuntu Server ISO.
  • Enable Skip Unattended Installation (manual install).
  • Allocate resources appropriate for the lab (example: 4GB RAM, 2–4 CPUs, 40GB disk).
  • Networking: NAT is acceptable for this runbook (SSH via port forwarding).

2) Ubuntu Server install (inside installer)

  • Create the user: sut.
  • Install OpenSSH server when prompted (or install after first boot).
  • Complete install. When prompted to remove installation media, eject the ISO (VirtualBox → Storage → Optical Drive → Remove disk).

3) Confirm network address (inside VM)

Expected: interface enp0s3 shows an IP (often 10.0.2.x on NAT).

4) Ensure SSH is running (inside VM)

sudo systemctl enable --now ssh
systemctl status ssh
ss -tlnp | grep :22

Expected: ssh.service active/running and port 22 listening.

9) Install SSH key (host → VM)

If ~/.ssh/id_ed25519 already exists on the host, reuse it. Do not overwrite.

12) Snapshot baseline

  • Create a VirtualBox snapshot named: baseline-ubuntu-sshkeys
  • Rule: snapshot before installing Docker, services, CI agents, or test stacks.
Private (operator): snapshot discipline note

Operator rule: if you ever have to “wonder what changed,” you missed a snapshot. Baseline snapshots are the lab’s equivalent of “golden AMI” discipline.

Common failure modes

SSH timeout

Usually indicates the host port forward is missing, wrong, or not bound. Check: lsof -iTCP:2222 -sTCP:LISTEN and confirm the VM is running.

Connection refused

Host is reachable but nothing is listening on 2222. Confirm the port-forward rule, then reboot the VM and re-check lsof.

Permission denied

Username mismatch (sut vs another user) or key not installed. Re-run: ssh-copy-id -p 2222 sut@127.0.0.1

Confusing host vs guest terminal

VM prompt typically looks like sut@sut-ubuntu:~$. macOS host prompt is different. Run host commands on the host.

Private (operator): rapid triage commands 
# Host side
lsof -nP -iTCP:2222 -sTCP:LISTEN || true
ssh -vvv -p 2222 sut@127.0.0.1 || true

# Guest side (inside VM)
sudo systemctl status ssh --no-pager | sed -n '1,18p'
sudo ss -lntp | grep -E '(:22\s)' || true

Why this baseline matters

  • Snapshots eliminate “unknown state” debugging and make experiments reversible.
  • Key-based SSH is the default access model for CI runners and lab automation.
  • With a baseline contract, every runbook that follows becomes predictable and testable.

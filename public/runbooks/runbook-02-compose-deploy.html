Runbook 02 — Explicit Targeting + Reachability (Sprint 2)

Enforce an explicit target (BASE_URL) and prove the system is reachable over the network by hostname. No defaults. No localhost illusions.

target discipline compose availability port exposure runner reachability

Objective + success criteria

Objective: Run tests only against an explicitly named, network-reachable SUT. Validate service availability locally on the SUT host and remotely from the runner via hostname.

Success criteria:

  • Target is explicit (no implicit defaults)
  • Target rejects localhost/loopback
  • Service ports are exposed on the SUT host (LAN-reachable, not loopback-only)
  • HTTP response is returned locally on the SUT host
  • HTTP response is returned remotely from the runner using SUT hostname
  • Dependencies are validated behind boundaries (DB verified from SUT, not exposed to runner)

✅ Sprint 2 — Completed (sanitized)

Sprint 2 is complete. The SUT is reachable by hostname from the runner, ports are explicitly published on the SUT host, and internal dependencies are validated strictly within the SUT boundary.

  • Runner DNS → TCP → HTTP reachability proven via sut.testlab
  • Ports 3000 (web) and 3001 (api) are LAN-reachable (not loopback-only)
  • DB dependency verified from inside the SUT only (mongosh ping returns { ok: 1 })
  • No implicit targets; no “localhost as environment” pattern

Failure modes + how to diagnose

Containers not running / crash looping

Symptom: docker compose ps shows exited containers.

Diagnosis:

cd ~/conduit
docker compose logs --tail=200

Port not listening

Symptom: no listener on expected port.

Diagnosis: verify published ports and container health.

cd ~/conduit
docker compose ps
docker compose port web 80 || true
docker compose port api 3000 || true

Local HTTP fails but port is listening

Symptom: connection succeeds but HTTP returns errors/timeouts.

Diagnosis:

cd ~/conduit
docker compose logs --tail=200

Remote HTTP fails but local HTTP succeeds

Symptom: SUT is healthy locally; runner cannot reach it.

Diagnosis (Runner):

nc -vz sut.testlab 3000
nc -vz sut.testlab 3001

Action: check firewall controls, port publishing, and network segmentation.

Why it matters (production relevance)

Availability validation reduces uncertainty before test execution. Target discipline ensures failures reflect real service or network conditions rather than incomplete startup or local substitution.

If a system is not reachable by name across the network, it is not production-real.

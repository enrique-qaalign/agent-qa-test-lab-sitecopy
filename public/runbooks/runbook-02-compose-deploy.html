Runbook 02 — Compose deploy (dev) + health check

Deploy the dev environment via Docker Compose on the SUT host and validate basic service availability from both SUT and runner perspectives.

compose up/down port exposure service availability runner reachability

Objective + success criteria

Objective: Deploy the dev SUT environment with Docker Compose and verify the service is available on the expected port and reachable over the network.

Success criteria:

  • Compose stack starts cleanly and containers are running
  • Service port is exposed on the SUT host (expected: 3001)
  • HTTP response is returned locally on the SUT host
  • HTTP response is returned remotely from the runner using SUT hostname/IP

Exact commands

Deploy dev stack (SUT host)

cd /path/to/sut-compose
docker compose pull
docker compose up -d

Verify containers (SUT host)

docker compose ps

Verify port is listening (SUT host)

lsof -nP -iTCP:3001 -sTCP:LISTEN || ss -lntp | grep ':3001' || netstat -an | grep '\.3001 '

Local HTTP check (SUT host)

curl -sS -o /dev/null -D - --max-time 3 "http://localhost:3001/" | head -n 12

Remote HTTP check (Runner)

export SUT_IP="192.168.0.31"
export SUT_PORT="3001"
export BASE_URL="http://sut.testlab:${SUT_PORT}"

curl -sS -o /dev/null -D - --max-time 3 "${BASE_URL}/" | head -n 12

Shutdown (optional, SUT host)

docker compose down

Sanitize hostnames and LAN IPs if publishing outputs.

Expected outputs (logs, screenshots, artifacts)

docker compose ps Containers show Up / healthy state (if healthchecks exist).
Port listening SUT host shows a listener on 0.0.0.0:3001 or the LAN interface, not loopback-only.
Local HTTP check Returns an HTTP/1.1 status line and headers.
Remote HTTP check Runner receives an HTTP/1.1 status line and headers from the SUT.

Failure modes + how to diagnose

Containers not running / crash looping

Symptom: docker compose ps shows exited containers.

Diagnosis:

docker compose logs --tail=200

Port not listening

Symptom: no listener on :3001.

Diagnosis: verify container ports and published ports.

docker compose ps
docker compose port <service-name> 3001 || true

Local HTTP fails but port is listening

Symptom: connection succeeds but HTTP returns errors/timeouts.

Diagnosis:

docker compose logs --tail=200
curl -sv --max-time 3 "http://localhost:3001/" || true

Remote HTTP fails but local HTTP succeeds

Symptom: SUT is healthy locally; runner cannot reach it.

Diagnosis (Runner):

nc -vz -G 2 192.168.0.31 3001
route -n get 192.168.0.31 | egrep 'destination|gateway|interface'

Action: check host firewall controls and network segmentation.

Why it matters (production relevance)

Deployment and availability validation reduces uncertainty before test execution. It ensures the system is reachable over expected ports and that failures in automation reflect real service or network conditions rather than incomplete environment startup.
